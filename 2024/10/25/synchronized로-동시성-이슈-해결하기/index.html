synchronized 로 동기화

Java 는 synchronized 블록을 사용해서 스레드의 작업을 동기화 하는 방법을 제공해줍니다. synchronized 라는 키워드를 사용하면 한번에 하나의 스레드만 접근 하는 것을 보장해 줍니다.

그럼 decrease 메서드에 synchronized 키워드를 사용해서 동시성 이슈를 해결해보도록 하겠습니다.



아니… 된다며! synchronized 키워드를 사용했지만 테스트 결과는 실패입니다. 동기화가 왜 안된 걸까요?

그건 바로 Spring 에서 제공하는 @Transactional 어노테이션 때문입니다. synchronized 와 @Transactional 을 같이 사용했기 때문이에요.

@Transactional 어노테이션을 사용하지 않고 synchronized 키워드만 사용해서 동기화 처리를 해보겠습니다.


동기화 처리에 성공했습니다. 동시성 이슈를 해결했습니다. synchronized 만 사용하니까 테스트에 성공을 하네요.

그런데 저는 이게 이해가 되지 않았습니다. synchronized 키워드는 한 스레드가 작업을 시작하면 다른 스레드들은 먼저 작업을 시작한 스레드가 작업을 종료할 때까지 대기함으로써 한 번에 하나의 스레드만 작업을 할 수 있도록 보장해 주는 것인데 왜 @Transactional 유무에 따라 결과가 달라지는 것일까요?

synchronized 와 @Transactional 의 상호작용

synchronized 의 동작 방식과 @Transactional 의 트랜잭션 관리 방식이 서로 영향을 미치기 때문에 먼저 작동 방식부터 살펴보도록 하겠습니다.

synchronized 키워드

synchronized 키워드는 스레드 간의 동시성을 제어합니다. 한 스레드가 메서드를 실행하고 있을 때 다른 스레드들은 대기하게 됩니다. 메서드가 종료가 되어야 다른 스레드가 이 메서드를 실행할 수 있습니다.

synchronized 키워드를 사용해서 동시성 문제를 해결할 수는 있지만 트랜잭션 범위가 어떻게 처리되느냐에 따라 다른 결과를 초래할 수 있습니다.

@Transactional 어노테이션

Spring 에서 @Transactional 어노테이션은 이렇게 작동합니다.

  1. 트랜잭션 시작
  2. decrease() 실행
  3. decrease() 종료
  4. 트랜잭션 커밋 혹은 롤백

@Transactional 은 메서드가 성공적으로 끝났을 때만 데이터베이스에 변경사항을 반영하고 메서드가 실패하면 작업 사항을 롤백합니다. 트랜잭션이 커밋되기 전까지는 변경된 데이터는 데이터베이스에 반영되지 않습니다. 다른 스레드에서도 변경된 데이터를 볼 수 없습니다.

메서드가 종료되면 변경 사항이 커밋되는 거 아니야?

저는 메서드 종료 시점이 커밋 시점이라고 생각했기 때문에 바로 이해가 되지 않고 의문이 남았습니다.

  • saveAndFlush() 메서드를 사용해서 변경된 데이터를 데이터베이스에 즉시 반영함
  • ‘}’ 를 만나서 메서드를 종료함
  • 메서드가 종료 되었기 때문에 대기하고 있던 메서드가 decrease() 메서드 작업을 시작함

제가 이해한 흐름은 이렇습니다. ‘한 스레드가 메서드를 종료하고 난 후 대기하던 스레드가 메서드를 실행하면 findById()로 조회한 데이터는 이미 변경된 데이터를 참조하는 것이니 동시성 이슈가 발생할 일이 없을 것이다.’

하지만 메서드의 종료 시점이 커밋 시점이 아니라는 것을 알게 되었습니다.

트랜잭션이 커밋되는 시점은 언제야?


스레드 A가 메서드를 실행하고 있고 스레드 B, C 는 A의 작업이 종료될 때까지 대기합니다. A가 데이터베이스에서 데이터를 조회해 올 때 quantity 는 100입니다. A가 quantity를 99로 수정하여 데이터베이스에 반영하고 메서드를 종료합니다.


스레드 A가 메서드를 종료했기 때문에 스레드 B가 메서드를 실행합니다. 하지만, 스레드 A의 트랜잭션이 아직 커밋되지 않았기 때문에 데이터베이스의 quantity는 변경되기 전의 데이터 100 입니다.

그렇기 때문에 스레드 B가 데이터베이스에서 데이터를 조회할 때 quantity 는 100이 되는 것이죠! 스프링이 트랜잭션을 커밋할 준비를 하고 있는 그 틈에 스레드 B가 decrease() 메서드에 접근해 데이터베이스의 데이터를 조회해오기 때문에 동시성 이슈가 여전히 발생하는 것입니다.

정리하고 넘어가자

  • @Transactional 어노테이션을 사용할 때 메서드가 종료되는 시점(‘}’ 를 만나는 순간)에는 아직 트랜잭션이 커밋되지 않았어요. 트랜잭션이 커밋되는 시점은 메서드 실행이 성공적으로 끝나고 난 후 Spring이 트랜잭션을 커밋할 때 입니다.
  • 그렇기 때문에 스레드 A가 decrease() 메서드를 실행하고 종료하는 시점에도 트랜잭션은 여전히 열려있는 상태 입니다. 트랜잭션이 커밋되기 전에는 데이터베이스의 변경 사항이 실제로 데이터베이스에 반영되지 않기 때문에 다른 스레드에서는 변경된 데이터가 아닌 변경 전의 데이터를 참조하게 됩니다.
  • 스레드 A와 B는 여전히 quantity 100을 읽게 되고 해당 데이터로 decrease() 메서드를 실행하기 때문에 동시성 문제가 여전히 발생하게 됩니다.

진짜 synchronized 로 동시성 이슈가 해결이 될까?

사실 synchronized 로 동시성을 해결하기엔 한계가 있습니다.

처리량 감소

모든 스레드에 lock 을 걸어버리기 때문에 한 번에 하나의 스레드만 작업을 수행합니다. 병렬 작업을 할 수가 없다는 의미인데요. 많은 요청이 들어오는 경우엔 한 스레드가 작업을 끝낼 때까지 나머지 스레드는 대기 상태에 놓이기 때문에 애플리케이션이 처리할 수 있는 처리량이 떨어집니다.

Deadlock 발생 가능성

synchronized 를 사용하게 되면 여러 스레드가 서로 다른 락을 획득하려고 할 때, 각 스레드가 상대방의 락을 기다리느라 무한 대기 상태에 빠질 수 있습니다.

서버가 두 대 이상이라면?

Java의 synchronized는 애플리케이션 레벨에서 적용됩니다. 하나의 프로세스 내에서만 보장이 됩니다. 서버가 한 대 일 때는 임계영역에 하나의 스레드만 접근할 수 있지만 서버가 두 대 이상일 때는 결국 여러 스레드가 접근할 수 있기 때문에 동시성 문제를 해결할 수 없습니다.

마무리하며

실제 운영 중인 서비스는 일반적으로 서버가 두 대 이상이기 때문에 현실적으로 synchronized 로 동시성 문제를 해결하는 것은 어려워보입니다. 이 문제를 해결하기 위해서 MySQL 에서 지원해주는 방법이 있는데요. 다음번엔 이 내용을 공부해서 정리해보고자 합니다.

강의 내용은 5분가량 이었지만 트랜잭션의 커밋 시점에 대해 잘못 알고 있어서 한번에 이해할 수 없었습니다. 이해하는 데 시간은 오래 걸렸지만 덕분에 몰랐던 부분을 확실히 짚고 넘어갈 수 있었어요. 동시성 문제를 해결하기 위해서 synchronized 키워드로 해결하거나 데이터베이스에 락을 거는 것 정도가 있다는 것밖에 몰랐는데 synchronized 와 @Transactional 이 어떻게 상호작용하는지를 알게 되었고, synchronized의 한계 때문에 데이터베이스 단에서 해결하는 방법으로 넘어가는 것까지 인과관계를 확실히 알 수 있었습니다.

그동안 저는 개발 공부를 할 때 이해가 안가는 부분은 찾아서 이해하고 바로 넘어갔습니다. 이렇게 글로 정리해 보는 시간은 갖지 않았어요. 정리하는 데 드는 시간 비용이 컸기 때문인데요. 이해한 내용을 글로 정리 하려고 하니 문장이 써지지가 않더라고요. “분명히 이해 했는데…? 그림으로는 설명할 수 있는데 왜 하나의 문장으로 설명이 안되지…?” 이해한 것을 다시 내 문장으로 표현 하는 과정에서 용어의 정확한 명칭에 대해서 배울 수 있었어요.

동기화, 임계영역 등의 키워드 하나로만 설명해도 듣는 사람이 한 번에 이해할 수 있는데 너무 주저리 설명하고 있었던 것을 깨달은 순간이었습니다. 이해만 하고 넘어가는 대신, 내용을 정리해서 설명해보는 시간을 가지니까 용어와 의미를 빠르게 전환하는 능력이 길러지는 것 같습니다.
