우테코 프리코스 2주차 회고


이번 프리코스 미션은 첫 시작부터 조금의 고비가 있었습니다. 과제를 할 수 있는 시간을 짜내서 계산해보니 대략 이틀 정도 나왔습니다. 첫주차 때 미션을 끝내고 다른 사람들의 코드를 보면서 세운 목표가 있었습니다. 2주차 때는 요구 사항을 더 꼼꼼히 분석 해서 최대한 많은 테스트 케이스를 작성해봐야겠다고 결심했었어요. 하지만 상황이 바뀌었고 그에 따른 목표를 다시 설정해야 했습니다.

주어진 시간이 많지 않으니 목표는 ‘기능 구현’ 이었습니다.

  • 최대한 빨리 기능 구현을 하자
  • 일단 돌아가는 쓰레기를 만들자
  • 하지만 테스트 코드 작성을 포기하지는 말자

아직 테스트 코드를 작성하는 습관이 생기지도 않았기 때문에 기능을 구현하면서 시간이 없다는 핑계로 제일 먼저 ‘테스트 코드 작성하기’ 라는 다짐부터 버릴 것 같았습니다. 하지만 테스트 코드를 작성하지 않으면 후반부에 가서 기능이 제대로 동작하는지를 확인하는 데 더 많은 시간이 소요된다는 걸 알기 때문에 테스트 코드를 작성하려고 노력했습니다.

아쉬운 점을 찾아보자!

이번 미션엔 아쉬운 부분이 굉장히 많은데요. 가장 크게 아쉬운 점 몇 개를 정리해보려고 합니다.

1. 예외 상황을 정리하지 못한 것

기능 구현을 목표로 세우다보니 예외 상황에 대해 깊이 생각해보지 못한 것이 가장 아쉽습니다. 과제 제출 마감 기한이 종료되고 다른 사람들의 코드를 리뷰할 때 분명 자책하게 될 지점이 될 것 같아요. 1주차 때 코드리뷰를 하면서 충격으로 와닿았던 것은 요구 사항에 대한 분석 부분이었습니다. 요구 사항에 대한 분석이 제대로 이루어지지 않으면 그게 예외 상황을 처리하는 곳까지 영향을 미친다는 것을 알았기 때문입니다.

다른 사람들은 어떻게 이렇게까지 생각할 수 있지? 설계하는 데 시간을 얼마나 할애하는 걸까? 그들의 결과물이 부럽기도 하고 ‘생각’을 어떻게 생각하는지 궁금하기도 했습니다. 2주차 때는 요구 사항에 대해 많이 생각해보려 했는데 이번에도 놓친 것 같습니다. Input/Ouput 을 처리할 때 많은 예외 케이스가 발생하는 것 같은데 아무리 생각해봐도 지금 처리해 놓은 것 외에 어떤 예외 케이스가 발생할지 아이디어가 떠오르지 않았습니다. 이번에도 다른 사람들의 코드를 보면서 배워가야 할 것 같아요.

2. 역시 급하면 테스트 코드부터 포기하는 것

TDD 로 시작했지만 중간에 포기했습니다. 마감 기한 내에 동작하는 코드를 만들지 못할 것 같았습니다. 회고를 작성하는 이 시점에 확인해보니 ‘레이싱 경주 기록’ 클래스와 ‘OutputView’ 클래스의 단위테스트를 작성 하지 못한 것에 대한 아쉬움이 큽니다. 기능을 구현하면서 애먹었던 부분이 두 클래스였기 때문이에요. 결과가 잘 찍혀 나오는지 확인해야 했는데 콤마(,)와 공백이 제거되지 않은 채 아래와 같이 출력되었습니다.

이 부분을 제대로 제거하고 코드가 제대로 잘 작동하는지 확인하려면 Application.main 메서드를 실행해서 사용자 입력값을 실제로 입력해야 했습니다. 단독 우승자가 나오면 확인이 되지 않기 때문에 공동 우승자 케이스가 나올 때까지 재실행 해야 했어요. 이때 후회했습니다.

“아…..테스트 코드만 있었으면 1초만에 끝나는데…”

성장한 부분을 찾아보자!

아쉽고 부족한 점이 많았지만 다른 사람과의 비교보다 나와의 비교가 성장을 이끌어내는 법이죠. 스스로 잘했다고 생각하는 부분도 있습니다.

1. 1주차의 피드백을 반영하다

1주차 때 입출력 부분을 뜯어내는 게 어려웠어요. 생각이 절차지향적으로만 되었기 때문인데요. 1주차 코드리뷰 시에도 이 부분에 관한 내용이 언급되었습니다. 다른 사람이 분리한 방식을 보면서 제 코드와 비교해보았고, 어떤 부분을 강하게 결합시켜놓았는지 찾을 수 있었습니다. 2주차 미션을 구현하면서 동일한 문제로 헤매게 되었지만 의존성을 분리해야 함을 인지하고 있으니 어느 지점에서 헤매고 있는지 발견할 수 있었어요! 입력 객체와 출력 객체에 각각의 역할을 부여하여 단일 책임의 원칙을 실천할 수 있었습니다.

그리고 1주차 코드리뷰 때 다른 분이 @ParameterizedTest 라는 키워드를 알려주셨었는데, 2주차 미션에서는 여러 개를 동시에 테스트 해봐야 하는 메서드가 많이 나와서 해당 키워드를 적용하여 해결할 수 있었습니다. 혼자 공부할 때는 그런 게 있는 줄도 모르고 넘어가는 일이 많은데 코드리뷰 때 이런 부분이 충족이 되니 앞으로도 더 열심히 코드리뷰를 주고 받아야겠다고 생각했어요.

2. 설레발일지 모르지만 글을 정리하는 습관이 생겨나려고해!

이 부분은 정말 생각지도 못했는데 좋은 습관을 득템할 수 있는 기회가 되고 있는 것 같습니다.저의 프리코스 미션의 모토는 “모르고 쓰지말자. 모르면 공부해서 내 걸로 만들고 넘어가자.” 입니다. 이 부분을 계속해서 상기하면서 코드를 작성하다보니 화면 한 구석에는 항상 노션이 켜져있습니다. 노션을 켜놓고 모르는 게 나오면 바로바로 정리하면서 스스로에게 꼬리질문을 해보고 있는데, 꼬리질문을 하다보면 막히는 지점이 나옵니다. 이 부분의 해결책을 찾기 위해서 다른 개발자분에게도 물어보고 스택오버플로우도 뒤지고 공식 문서도 찾아보니, 정리하면서 생각이 구체화되는 경험을 하고 있습니다. 이것저것 막 찾다보면 “잠깐, 그래서 내가 지금 알고 싶은 게 뭐지? 뭐 찾고 있었지?” 하는 순간이 있는데 정리를 하면서 따라가다보면 명확해지더라고요. 이렇게 공부하는 게 도움이 되는 거 같다고 생각은 했는데 다른 개발 공부를 하면서도 글을 정리하고 있는 걸 보니 꽤나 뿌듯해집니다. 항상 다짐을 할 때 버킷 리스트에 제일 먼저 적는 게 ‘공부한 거 정리해서 블로그에 올리기’ 였거든요. 이번엔 이루어질 것 같다는 생각이 듭니다.

한 주를 돌아보며

회고를 작성하기 전의 제 머릿속 상황은 이렇습니다. “아… 이번주는…망했다.” 코드를 너무 급하게 짠 것 같았고 만족스럽지가 않았기 때문입니다. 부족하고 아쉬운 부분으로만 채워졌었어요. 이번 주엔 성장을 못했나? 아무 것도 얻은 게 없나? 회고를 하면서 차근차근 되짚어보니 얻은 게 있었습니다. 남과의 비교 말고 저번주의 나와 비교를 하니 그제서야 보였습니다. 어떤 부분을 수용했고 또 어떤 변화가 있었는지를요. 2주차 미션을 끝내고 회고를 하면서 제가 배운 부분은 ‘나한테 집중하자’ 입니다. 어제는 A를 몰랐는데 오늘은 A를 알게 됐다면 저는 오늘 하나를 더 알게 된 사람이니까요.

“잘하는 사람들을 보면서 모방하자. 그들을 따라가기 위해서 할 수 있는 모든 것을 하자. 나중에 뒤에가서 후회하지 말자. 그리고 나 스스로를 깎아내리지 말자” 이런 생각들을 해보았습니다.
2주차 미션도 열심히 코드리뷰 해서 3주차 미션 때 더 많이 성장해 있을 모습을 상상하며 2주차 회고는 여기서 마치겠습니다. 🙋🏻‍♂️
