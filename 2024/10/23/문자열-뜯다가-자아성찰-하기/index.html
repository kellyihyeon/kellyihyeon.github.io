문자열 뜯다가 자아성찰 하기

프리코스 1주차 미션을 진행하면서 제 자신에 대해 파악한 것이 있습니다. String 객체가 제공해주는 기본 메서드를 그동안 그냥 막 쓰고 있었다는 것이에요 😭 아래의 기능 요구 사항을 보면 주어진 문자열에서 구분자와 숫자를 추출해야하는 기능이 필요하다는 것을 알 수 있습니다. 우선 제가 하고 싶었던 것이 있었는데요. 문자열에 잘못된 구분자가 포함되어 있는지를 검사하고 싶었어요.

저는 커스텀 구분자를 지정하는 경우에는 커스텀 구분자’만’ 사용 가능하게 해야 한다고 분석하고 기능을 구현했습니다. (하지만…기본 구분자 + 커스텀 구분자를 사용하는 것으로 해석하는 것이 맞는 것 같아요…흑 요구사항부터 잘못 분석했습니다.) 이 글에서는 제가 분석한 것을 토대로 설명하겠습니다.

다시 돌아가서! 기본 구분자로는 **콤마(,)와 콜론(:)**만을 사용할 수 있습니다. 아래의 예시 코드는 기본 구분자 문자열로 주어진 input 값입니다. 예를 들면서 설명해보겠습니다. 아래와 같은 문자열이 주어졌을 때 input 값에 잘못된 구분자가 있으면 예외를 발생시키려고 합니다.

1. split 하면 바로 해결 가능 하잖아!

이렇게 split 을 하면 결과는 split = [1, 2, 3] 이렇습니다. 아주 간단합니다. 기본 구분자를 기준으로 숫자만 추출되었어요. 하지만 다른 구분자가 섞여있는 input 값은 어떨까요?

결과는 이렇습니다. wrongInput = [1, 2, 3*4]

추출된 숫자는 1과 2 그리고 3*4 입니다. split 메서드는 콤마(,)와 콜론(:)을 기준으로 분리를 해주었기 때문에 당연한 결과 입니다. 의도한 대로 결과가 나오게 하려면 split 메서드는 적절한 메서드가 아닙니다. 구분자가 잘못된 것인지 판단을 하려면 구분자로 사용된 * 를 구분자로 떼어낼 수 있어야 합니다.

미션을 진행하면서 생각의 흐름을 기록하기 위해 아주 날 것 그대로 글을 써내려 갔습니다.
그리고 지금 여러분이 보고 계시는 이 글은 정리가 (잘) 된 글이고요. 저는 지금…글을 정리하면서 과거의 제 머릿속을 마주하고 있는데요. A를 하려면 B를 해야해! 라고 깨우쳤다가 왜 갑자기 B는 안하고 C를 하고 있는지 모르겠습니다… 네, 바로 다음에 나오는 작업이 B가 아니고 C 이기 때문에 이렇게 말이 많아졌습니다…

2. 그럼 다른 메서드를 써보자!

하고 싶었던 것은 문자열에 콤마와 콜론이 아닌 다른 구분자가 포함되어 있는지 여부를 판단하는 것이었습니다.
split 이 아니라 다른 메서드를 사용해야 할 것 같은데… 그때 contains 메서드와 matches 메서드가 떠올랐습니다. 일단 문자열에 콜론이 있는지 테스트 해보기로 했어요. 포함되어 있다고 나온다면 반대 표시를 붙여서 콤마(,)와 콜론(:)이 아닌 것이 있는지를 검사하면 바로 나올 수 있을 거라고 판단했기 때문입니다.
이렇게요. if(wrongInput != null) 이렇게 반대로 해석하게 하면 되겠지? 라고 생각했어요.

… 이게 뭐야… matches 는 왜 저런 결과가 나온 걸까요? 콜론이 분명히 있는데 왜 false 로 나오는 걸까요? 일단 시간이 없으니까 나중에 찾아보고 contains 로 해결해보기로 했어요.

콜론 하나로 테스트 해보니 잘 나오는데 그럼 이제 콤마까지 파라미터로 전달하려면 어떻게 해야할지를 생각해봤어요. “basicDelimiter 를 정의한 것처럼 하면 되겠지?” split 메서드를 사용해서 기본 구분자를 기준으로 숫자를 추출했듯이 말이에요.

split 메서드를 정의한 것처럼 |(또는) 연산으로 연결해주면 될 것 같았는데 false 가 나왔습니다. or 연산이 적용되어서 콤마 또는 콜론이 포함되어 있는지를 검사해줄 것 같았는데 아무래도 ,|: 라는 문자 자체를 검사해서 결과가 false 로 나온 것 같았어요. wrongInput 에는 ,|: 라는 문자는 없으니까요.

테스트를 해보니 맞습니다. ,|: 를 하나의 문자로 인식을 하네요. 음, 그렇다면 문자열 안에서 or 연산을 인식하게만 하면 끝날 것 같았습니다. 어떻게 정의해야 or 연산으로 인식할 수 있을까요? 우선 contains 메서드의 파라미터는 어떻게 정의되어 있기에 split 은 되는데 contains 는 안되는 건지 알고 싶었습니다. 문서를 확인해 보기로 했어요.

2.1. split 메서드

split 메서드는 파라미터로 정규표현식을 전달해야 합니다. 알고리즘 문제를 풀 때 자주 썼던 메서드라 음..그렇지.. 하고 넘어갔어요.

2.2. contains 메서드

…어??? contains 메서드의 파라미터는 정규표현식이 아닙니다…😭 이제야…contains 메서드 테스트 결과가 납득이 갑니다…그러면 contains 메서드로 정규표현식을 전달할 수 없습니다. 전달해봤자 정규표현식은 더 이상 정규표현식이 아닌 게 되겠죠… contains 메서드를 사용해서 풀 수 없는 문제임을 인지했습니다.

2.3. matches 메서드

혹시나 싶어서 matches 메서드를 다시 소환합니다. Yeah!!! matches 메서드는 정규표현식을 파라미터로 전달받습니다. “아? 내가 원하는 걸 하려면 contains 로는 안되고 matches 를 써야되는구나!” 그럼 matches 를 테스트 해보겠습니다.

나한테 왜그래… 정규표현식을 파라미터로 전달해줬잖아… 왜 false 라고 나오는 거야… 😢

input 에는 분명히 콜론이 있는데 왜 false 라고 뜨는 걸까요… 정규표현식을 정의하는 규칙이 있나? 그걸 깨뜨리고 잘못된 방법으로 정의했나? 정규표현식을 작성하는 방법을 찾아보았습니다.

정규표현식 문법을 참조하여 만들어보았고 테스트는 통과했습니다. 참조한 링크는 여기에 공유해두겠습니다!

4. 방법을 다시 찾자!

네, 포기했습니다. 일단 과제를 구현하는 것이 0순위 이니까요. 주어진 시간 안에 해야할 것들이 너무 많은데 더 붙잡고 있을 순 없다고 판단했습니다. 그래서 다른 방법으로 구현해보기로 했어요. 이번 미션에서 선택한 방법은 아래와 같습니다.

결과적으로 좀 둘러가게 되어서 아쉽지만 지금 이 감정도 과정을 기록해두었기 때문에 느낄 수 있는 것이라 생각해서 감사하게 여기려고 합니다.

4.1. 문자열에서 구분자만 남긴다

4.1.1. 기본 구분자를 사용한 경우

여기 input 값이 있습니다. “1:2.3”

요구 사항을 살펴보면 입력값은 구분자와 양수로 구성된 문자열 임을 알 수 있습니다. 소수도 포함이라는 얘기죠. 문득 이런 생각이 들었습니다. “온점(.)을 구분자라고 인식하면 어떡하지?” 사용자는 2.3 이라는 소수를 전달한 거라면 연산에 필요한 양수는 1과 2.3이 되어야 합니다. 하지만 온점을 구분자로 인식하면 1,2,3 이라는 양수가 추출될 거예요.

그래서 문자열에서 양수를 떼버리고 구분자만 남기기로 했어요.

양수를 추출하는 표현식은 정규식 문법을 참고하여 만들었습니다. 여기 참조 자료가 있습니다!

양수인지 확인하는 패턴은 만들었으니 이제 문자열에서 구분자만 남기는 걸 해보겠습니다. 양수인 것을 공백으로 치환하는 방법을 사용하기로 했어요.

4.1.2. 커스텀 구분자를 사용한 경우

커스텀 구분자를 사용한 문자열은 여기 있습니다. "//;\n1;2;3" 우선 커스텀 구분자인지 확인하기 위해 개행 문자로 잘랐을 때 잘라지는지 체크해보기로 했어요.

이제 커스텀 구분자를 사용한 문자열도 문자열에서 구분자만 남겨보기로 합니다.

처음에 테스트 코드부터 짜면서 확인하느라 코드가 좀 복잡하지만, 의도한대로 결과가 나오고 있는 걸 확인할 수 있습니다.

4.2. 남아 있는 구분자에서 타입별 구분자를 제거한다

기본 구분자를 사용한 경우에는 남아 있는 구분자에서 콤마(,)와 콜론(:)을 제거합니다. 커스텀 구분자를 사용한 경우에는 사용자가 정의한 구분자를 제거해봅니다.

4.3. 잘못된 구분자가 있으면 예외가 발생한다

문자열에서 구분자를 제거한 결과 result 값이 비어있는지 확인합니다. 위에 작성한 두 개의 테스트는 모두 성공합니다. 잘못된 구분자를 사용한 경우 IllegalArgumentException 예외가 발생하도록 코드를 추가로 작성하였습니다. validate 메서드에서 4.2. 의 로직을 수행하고 있어요!

5. 번외: matches 메서드로 해결하는 방법을 알아내다

과제를 제출하고 나서 이 부분을 다시 찾아보았어요. 해결책은 의외로 간단했습니다. 정규표현식 만드는 방법을 좀 더 꼼꼼히 읽어봤어야 했는데 말이에요 ㅠ_ㅜ……

문자열이 , 와 : 만 포함하고 있는지 검사

다시 돌아가보겠습니다!

resultB 에서 사용한 ".*:.*|.*,.*" 이 패턴은 문자열에 , 와 : 가 포함된 경우를 검사하고 있어요. 그래서 $가 섞여 있음에도 당연히 true 가 나옵니다. 제가 원하는 결과는 , 와 : 외에 다른 문자가 있으면 false 가 나오게 하는 것이었어요. 그래서 [,:] 이 문법을 사용했고 실패했었죠.

그런데… 문자열에 , 와 : 만!!! 포함되어 있는지 검사하는 정규식은 다음과 같아요.

resultC 를 봐주세요. 핵심은 + 입니다. 정규식 패턴 ‘+’를 사용해서 0-9와 콤마 그리고 콜론이 1개 이상인 것을 허용했습니다.

  • [0-9] 는 0부터 9까지의 숫자를 허용합니다.
  • [,] 는 콤마를 허용합니다.
  • [:] 는 콜론을 허용합니다.
  • + 는 대괄호 안에 있는 문자들이 하나 이상 반복될 수 있다는 의미입니다.

머리가 아프니까 요약을 해보자

  • ".*:.*|.*,.*"

    • 문자열에 콜론 혹은 콤마가 있는지 체크합니다. 그렇기 때문에 문자열에 다른 구분자가 섞여있더라도 결과는 TRUE 입니다.

  • "[0-9,:]"

    • 문자열에 0부터 9까지의 숫자, 콤마, 콜론이 있는지 체크합니다. 얼핏 생각하면 이 패턴도 맞는 거 아닌가? 라고 생각할 수 있지만 아래의 결과를 보시죠.

    • inputB 를 보면 구분자는 콤마와 콜론으로만 이루어져있지만 결과는 FALSE 입니다. 콤마가 1개 콜론이 2개이기 때문입니다.

  • "[0-9,:]+"

    • 문자열에 0부터 9까지의 숫자, 콤마, 콜론이 1개 이상이 있는지 체크합니다.
    • inputB 를 보면 콜론이 2개가 있지만 결과는 TRUE 입니다. inputC 에는 $ 가 섞여 있기 때문에 결과는 FALSE 로 나옵니다! 드디어 ㅠㅠ !!!

메서드를 어떻게 사용해야하는지에 대한 가이드가 자세히 적혀있는 문서가 있는데, 결과가 이상하게 나올 때 바로 문서를 찾아보진 않았어요. 블로그를 찾아보거나 스택 오버플로우를 뒤져서 해결했었어요. 생각해보면 이게 제 코딩 습관인 것 같아요. 남(?)한테 의지하는 것… 이번 기회에 바꿔보기로 했습니다. 앞으로 모르는 게 나오면 문서부터 먼저 확인해보자! 확인해봤는데도 이해가 안가면 그때 블로그를 찾아보자!

되게 간단한 기본 지식을 이렇게 길게 정리 하면서도.. 아….이게 맞나…. 여태 이걸 모르고 썼다는 게 말이 되나… 부끄럽게만 느껴졌습니다. 그렇지만 이렇게 잘 정리하고 넘어가면 다음번에는 좀 덜 헤매지 않을까 생각합니다!
