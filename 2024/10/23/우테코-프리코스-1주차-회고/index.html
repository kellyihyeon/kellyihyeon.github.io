프리코스 1주차 회고

문자열 덧셈 계산기..? 더하기 기능만 있는 계산기를 만들면 되는 건가?

기능 요구 사항을 읽고 처음 든 생각은 “뭐지..? 쉬…..운 거 아니야….? 빨리 끝날 것 같은데?” 였습니다. 우선 기능 개발에 앞서 요구 사항 정의서를 작성하기로 했어요. 프리코스 과정을 시작하기 전에 만들었던 프로젝트에서 배운 점이 있었거든요. 그 프로젝트에서는 요구 사항 분석 및 정의서를 작성하지 않고 바로 기능부터 개발하는 바람에 진행하면서 혼란을 겪었어요. 요구 사항을 명확히 정의해놓질 않으니 설계가 계속 변경 되었거든요. 요구 사항 정의서의 중요성에 대해 배울 수 있었던 값진 시간이었습니다.

배운 거 써먹기

그리고 이 경험을 문자열 덧셈 계산기에 바로 써먹기로 했습니다. 이번 주차에서 도전해 보고 싶은 것들을 해보기로 했어요.

  1. 요구 사항 정의서 작성하기
  2. 주어진 시간의 반 이상을 설계에 투자할 것
  3. TDD 로 코드 작성해 보기 (TDD 가 너무 막막하다면 테스트 코드를 반드시 작성할 것)

이 세가지는 평소의 개발 패턴과는 완전 달랐어요. 정확히 말하면 완전 반대라고 말할 수 있습니다. 요구 사항 정의서를 작성하지 않고 바로 개발부터 들어가기, 코드를 작성하면서 비즈니스 로직 생각하기, TDD 는 커녕 테스트 코드도 작성하지 않기…

1주차에 비교적 쉬운 문제가 나온 것이 오히려 기회라고 생각했습니다. 제가 어려워 하는 것들을 이번 주차에 도전해볼 수 있었으니까요. 그리고 프리코스를 진행하면서 스스로 세운 규칙이 있는데, ‘ChatGPT 절대 사용하지 않기. 변수명, 클래스명 짓는 것조차 스스로 해결할 것’ 입니다.

이번엔 어떤 경험을 했을까요?

미션을 진행 하면서 스스로 세운 목표를 지키려고 노력했고 미션을 마무리 한 지금 목표를 전부 이뤘습니다! 이번 미션을 진행하면서 깊게 와닿았던 부분은 TDD 와 객체지향이었어요.

TDD 그거 나도 할 줄 알아!

요구 사항 정의서를 보면서 TDD 로 하나씩 하다보니 구현해야 할 영역의 크기가 저절로 작아졌다는 것이었습니다. 평소 코드를 작성하면서 답답했던 지점이 이 부분이었어요. 구현해야 하는 서비스의 모든 것이 머릿속에 있다보니 A를 만들면서 B가 튀어나오고 C와의 관계를 같이 생각하다보니 결국 A의 영역이 커졌었거든요.

그런데 테스트 코드를 먼저 작성하면서 내가 지금 구현해야 할 기능만 써내려가고 이 기능을 통과시키려면, 초록불이 들어오게 하려면 어떻게 해야하는지에 대해서만 생각하면 되니 코드가 간결해졌습니다.

TDD야 고마워

TDD로 이득을 본 것은 여기에서 끝나지 않았습니다. 리팩토링을 할 때도 느낄 수 있었어요. 하나의 예를 들어보자면, 입력 요구 사항인 양수에 대한 상수를 외부에서 선언해놓고 사용하고 있었어요. 그런데 이 상수는 DelimiterExtractor(구분자 추출기)에서 필요한 상수였거든요. 뒤늦게 이 상수를 해당 클래스의 필드에 선언해놓아야겠다고 판단했습니다. 그런데 문제가 생겼어요. 해당 클래스 내에서는 이 상수를 파라미터로 전달받는 메서드들이 다수 있었어요.

“음… 만약 이 메서드들을 수정하면…어느 범위까지 수정해야 하는 걸까? 메서드를 사용하고 있는 모든 곳을 다 고쳐야할텐데…” 과제 제출일 마지막 날에 리팩토링을 하다가 되던 기능도 안돼서 테스트 통과도 못하면 어떡하지 싶었습니다. 뭐가 됐든 미션을 구현하는 것이 0순위 였으니까요. 리팩토링 하지말고 그냥 둘까 고민했습니다. 불안했어요. 그런데 갑자기 테스트 코드가 생각이 났습니다. “아!? 잠시만, 나 테스트 코드 짰잖아????!!”

테스트 코드가 있으니까 일단 수정해보고 테스트를 돌려보면 어느 곳을 고쳐야 하는지 확인할 수 있으니 일단 확인만 해보자는 생각에 빠르게 리팩토링을 했고 테스트를 돌렸습니다. 생각했던 것보다 수정할 곳은 적었고 실패한 테스트 코드의 콘솔을 보니 어느 부분을 말하는 건지 바로 알 수 있었습니다.

테스트 코드 덕분에 실패 지점을 정확히 알 수 있었고 하나하나 찾아가면서 고치니까 30분도 안걸려서 끝이 났어요. 전체 테스트 코드를 실행했을 때 전부 초록불이 뜨는 것을 확인하고 처음엔 좀 얼떨떨했습니다. “뭐야…이게 되네..?… 이래서 테스트 코드를 짜야한다고 하는 거구나…”

미션을 시작하기 전에 스스로 목표를 설정했지만 하루하루가 지날수록 불안했어요. 디스코드를 보면 하루 이틀만에 끝낸 사람들이 많이 보여서 불안했습니다. “목표 지키면서 하느라고 기한 안에 못끝내면 어떡하지?” 이런 불안감이 들 때마다 지금 배우고 있는 것, 새롭게 얻게 된 깨달음에 집중했습니다.
TDD와 테스트 코드를 작성하는 것에 대한 중요성을 이론적으로 알고는 있었지만 테스트 코드까지 작성할 시간은 없다는 핑계로 항상 테스트 코드는 미뤄두고 있었거든요.

이제 저한테 테스트 코드는 믿는 구석이 되었어요. 커밋 하기 전에 테스트 코드를 한번에 다 돌려보고 깨지는 부분이 있는지 확인할 수 있었으니까요. 실행 결과가 초록색이라는 건 방금 수정한 비즈니스 로직이 다른 곳에 영향을 미치지 않았다는 것을 의미하니까 이걸 믿고 마음 편하게 코드를 작성할 수 있었습니다. 이번 미션은 테스트 코드를 작성할 수 있었던 감사한 시간으로 남게 됐습니다!

객체지향은 여전히 어렵다

뿌듯하고 자랑스러운 시간이 있었다면 당연히 아쉬웠던 점도 있었는데요. 객체지향적으로 생각하는 부분이 여전히 어렵습니다. 요구 사항 정의서를 보면서 기능을 구현 했지만 여전히 어느 객체의 책임인지 모호한 순간들도 있었습니다.

DelimiterExtractor(구분자 추출기)와 NumberExtractor(숫자 추출기)가 각자의 역할과 책임을 지고 있었는데 문자열을 파싱해야 하는 역할은 누구의 역할인지 헷갈렸습니다. 이런 순간을 맞닥뜨리면 객체들이 머릿속에서 또 엉켜버립니다. 애초에 이렇게 분리한 게 잘못된 것인지, 다른 방법으로 풀었어야 했는지, 다른 사람들은 어떻게 해결했을지…

결국엔 StringParser(문자열 파서) 라는 새로운 객체가 이 역할을 수행하기로 결론 지었지만, 확신을 할 수가 없었습니다. 어떻게 해야 객체지향적으로 생각할 수 있는지 조금 답답한 마음이 남았습니다.

알고 사용하자

그리고 이번에 알게되었던 사실은 String 의 기본 메서드의 쓰임새를 잘 모르고 사용하고 있었다는 점입니다. String 객체를 가지고 이것저것 해보면서 split(), matches(), contains(), replace(), replaceAll() … 이런 기본적인 메서드들을 사용해 볼 기회가 많았어요.

그런데 어떤 메서드에서는 의도한 대로 결과가 나오는데, 왜 어떤 메서드에서는 의도한 대로 되지 않는지 찾아보면서 java doc 을 찾아보는 시간을 많이 가졌습니다. java doc 을 읽으면서 알게 되었어요. 파라미터의 타입이 다르다는 것을요. 기본적인 가이드를 스킵하면서 코드를 작성하고 있었음을 알게 된 감사한 순간이었습니다.

1주차가 끝난 지금

일주일이라는 시간이 빠르게 지나갔고, 미션을 진행하면서 배운 점이 많습니다. 아쉬웠던 점이 기억에 많이 남아있지만 덕분에 어떤 점이 부족한지 알 수 있게 돼서 감사한 시간이었습니다. 알아차리고 깨닫게 됐으니 이 부분은 보완해나가면 된다고 생각해요.

다음 미션을 진행하면 또 다른 형태의 아쉬운 점이 나오게 될텐데 오히려 좋은 것 같아요! 어떤 게 부족한지 계속 발견할 수 있으니 성장할 수 있는 기회를 가지는 거라고 생각합니다. 다음 미션도 목표 설정을 잘 해서 할 수 있는 걸 전부 다 해봐야겠어요!

번외: 좌절과 극복의 사이클

좌절의 순간

네, 과제를 제출하고 프리코스 커뮤니티 활동을 통해서 코드리뷰 해보는 시간도 가졌는데요. 몰래(?) 다른 사람들의 코드를 읽으면서 충격을 받게 됩니다…
제 코드에 빈틈이 너무 많다는 사실을 알게 되었어요. 예외처리를 꼼꼼히 하지 않았다는 거죠. 제 코드가 너무 부끄럽게 느껴졌어요. 코드 짜면서 분명히 아…이건 진짜 잘 짰다.. 핫… 하는 부분도 있었거든요. 코드를 엎어버리고 싶었습니다.

요구 사항 분석부터 잘 못했다는 사실을 알았을 땐 좌절 했습니다.ㅠㅠㅠㅠㅠ 저는 사용자가 커스텀 구분자를 등록한 경우에 커스텀 구분자만 사용하게 해야 한다고 이해하고 이를 토대로 정의서를 작성했거든요. 그런데 다른 사람들 코드를 보고 기능 요구 사항을 다시 찬찬히 읽어보니 ‘기본 구분자 + 커스텀 구분자’를 혼용해서 사용하게 해야하는 게 맞다는 생각이 들었어요.

문자열을 파싱하는 것도 더 간단하게 자르는 방법이 있었는데, 왜 복잡하고 어려운 방법을 선택했는지도… 아니 선택이 아니라 그 방법밖에 떠오르질 않았으니 실력 부족입니다.

이제 알았으니까 괜찮아

좌절 다음은 극복입니다. 저는 항상 이런 사이클을 반복하면서 성장하는 유형의 사람입니다.🔥🔥🔥

용기내서 디스코드 채널에 코드리뷰 댓글을 달았습니다.


나도 코드리뷰 해봤어!

코드 리뷰를 안해봐서… 뭐 어떻게 하는 건지..




소소하지만 일단 발견한 오타를 알려드리는 것부터 해봤습니다. 읽으면서 제가 이해한 게 맞는지 다시 물어보기도 하고 다른 방법도 제안해보기도 했습니다. 아… 이게 뭐라고 이렇게 떨리죠…

내 코드에도 리뷰가 달렸어!

정말 감사하게도 코드리뷰를 하면서 많이 배웠습니다. 제 코드에 달린 리뷰를 보면서 정말 생각이 많아졌습니다. 혼자 코드를 작성할 때는 인지하지 못했던 것들을 다른 사람들의 질문을 통해서 알게 되었어요. 전혀 알지 못했던 개념에 대해서도 키워드를 얻을 수 있었구요. 이래서 코드리뷰 하는 거구나ㅠㅠㅠ
2차 미션 제출하고 나면 코드리뷰를 좀 더 많이 해보려고 해요!

번외: 잘 안되는 것과 알아봐야 할 것

잘 안되는 것

과제를 풀면서 사용자에게 보여줘야 하는 화면이 객체로 분리가 잘 안됐었어요.
과제를 진행하는 당시에 작성해놓은 메모를 보면 이렇습니다.

아래의 사항들이 마음에 걸렸습니다.

  1. input 값을 그대로 쓰는 게 아니라 parseBr 로 해줘서 써야 하는 것
  2. number 추출기에 input 을 넘길 때는 또 연산식으로 변환해 줘서 넘겨야 하는 것
  3. validate 할 때 delimiter 에서는 양수패턴을 파라미터로 넘겨야 하는 것
  4. validate 할 때 number 에서는 구분자 정규표현식을 파라미터로 넘겨야 하는 것
  5. 숫자를 추출할 때도 정규표현식을 넘겨야 하는 것

이런 사항들을 단번에 파악할 수 없을 것 같다는 생각을 했습니다. 파라미터로 regex 를 전달해야 하는데 어떤 정규표현식을 전달해야 하는지 정리가 된 문서가 없으면 코드만 읽고서는 혼란스러울 것 같다고 생각했어요.

알아봐야 할 것


new 를 통해서 객체를 계속 생성하게 되는 것과 static 을 사용하는 것에 대한 장단을 알 수가 없어서 저런 의문이 들었어요. 그래서 객체를 생성하는 방법으로 코드를 구현했습니다.


그랬더니 이런 리뷰를 받게 되었습니다! 과제 제출하고 나서 알아봐야지~ 하고 까먹고 있었는데 이참에 공부하고 넘어가야겠어요.

이제 진짜 마무리

정말 모르는 거 투성이네요… 그런데 이렇게 모르는 게 계속 나온다는 게 또 개발의 매력이라고 생각해요. 지루할 틈이 없으니까요! 1주차 때 실수하고 많이 놓친 부분은 2주차 미션을 진행하면서 보완하려고 합니다. 그럼 이제 진짜 마무리를 해보도록 하겠습니다. 🙋🏻‍♂️
