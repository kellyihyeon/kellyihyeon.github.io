동시성 이슈가 뭐죠?


어느 날 이런 문구를 접하게 됩니다. 저도 겪어봤던 상황이었어요. 전에 다녔던 회사가 이커머스 플랫폼이었거든요. 데이터베이스에 저장된 상품의 재고 수와 실제 상품의 재고 수가 안맞거나 배송 상태가 일치하지 않는 상황이 꽤 자주 있었습니다. 팀에서는 이 문제를 해결하기 위해 데이터베이스에 쿼리를 직접 날려서 수정하거나 관리자가 수동으로 업데이트를 하곤 했습니다. 해결 방법이라고 하기엔 적절하지 않았어요.

그 당시에 저는 이런 생각을 했었어요. “이제 갓 취업한 신입인 내가 저 문제를 주도적으로 해결할 수 있을까..? 시니어 개발자분들이 해결하시겠지?” 회사는 아주 바빴고 리팩토링을 할 시간도, 테스트 코드를 짤 시간도 없이 기능 개발을 해야만 했습니다. 기획팀과 마케팅팀에서는 계속해서 새로운 기능을 요청했고 개발팀은 기능을 구현하기 위해 코딩 코딩 또 코딩을 해야 했어요.

그런데 반복해서 발생하는 문제를 바쁘다는 핑계로 미뤄둔 건 아니었나라는 생각이 들기도 합니다. 원인을 파악하고 해결책을 제시해서 문제를 해결했더라면 어땠을까요? 현업에서 일하는 분들이 좀 더 편하게 업무에 집중할 수 있지 않았을까요? 저 문구는 인프런의 어느 한 강의의 소개에 나와있던 걸 들고와봤습니다. 그때의 문제 상황을 다시 정리해보면서 원인 파악과 해결책을 찾아보려고 해요.

1. 무엇이 문제였을까?

어떤 주제를 공부할 때 제일 빠르게 배울 수 있는 방법은 직접 그 상황에 놓였을 때라고 생각해요. 강의를 들으면서 여기에서 배우는 지식을 체화해서 내 것으로 만드려면 진짜 케이스 속으로 들어가야 한다고 생각했어요. 그래서 그 당시에 데이터 정합성에 문제가 생겼던 케이스를 정리해보았습니다.

  1. 오프라인 매장 포스기와의 동기화 지연
  2. 외부 API의 응답 지연
  3. 포인트 적립/차감
  4. 결제 시 재고 확인
  5. 배송 상태 업데이트

문제가 발생했던 여러가지 상황 중 1번 같은 상황은 데이터 동기화 지연 문제이고, 2번 같은 상황은 주로 네트워크 불안정 문제입니다. 3번 ~ 5번 상황으로 시나리오를 만들어볼 수 있을 것 같아요.

2. 동시성 이슈

동시성 이슈는 하나의 프로세스에서 여러 스레드가 동일한 자원을 공유할 때 발생하는 문제를 의미합니다. 임계영역에는 한 번에 하나의 스레드만 접근할 수 있도록 보장해야 하는데 이것이 지켜지지 않을 때 문제가 발생하게 됩니다.
어떤 상황을 말하는 것인지 사례로 설명해보도록 할게요.


재고 도메인은 ID, 상품ID, 재고수 데이터를 가지고 있습니다. 재고 도메인의 decrease 메서드에서는 재고를 감소 시키는 역할을 하고 있어요.


재고를 감소시키는 서비스 로직입니다. 문제가 되는 상황을 가정해봅시다. 어떤 상품의 재고가 100개가 있습니다. 만약 동시에 재고 감소 요청이 100개가 온다면 어떻게 될까요? 우리가 원하는 결과는 이 제품의 재고 수가 0개가 되는 것입니다. 원하는 결과대로 될지 테스트를 해보겠습니다.


0개를 기대했지만 재고 수가 90개 입니다. 왜 이런 일이 발생한 걸까요?

stock(id=1)의 초기 재고수는 100개 입니다.
stock 을 조회해와서 재고 수를 1개씩 감소시키는 작업이 순차적으로 100번 반복된다면 재고 수는 0이 될 것입니다. 하지만 해당 테스트에서 100개의 스레드를 생성했고 이 스레드들은 순서를 보장받을 수 없습니다. 여러 개의 스레드가 stockService.decrease() 메서드를 실행하게 됩니다.


여러 스레드가 decrease 메서드에 접근하여 1번 데이터를 조회 합니다. 이때 스레드들은 1번 stock의 quantity 를 100으로 읽게 되고 decrease 로직을 수행하면 quantity 가 99가 됩니다. 임계영역인 quantity 에 여러 스레드가 접근하였기 때문에 동시성 이슈가 발생하게 된 것입니다.

3. 동시성 이슈가 발생하는 케이스를 살펴보자

동시성 이슈가 무엇인지 이해하기 위해 간단한 재고 케이스로 설명을 드렸는데요. 동시성 이슈가 발생하는 다른 케이스들을 살펴보겠습니다.

3.1. 결제 시 재고 확인

재고가 1개인 상품이 있습니다. 사용자 A와 B가 동시에 이 상품을 결제하려고 하면 실제로는 한 명의 사용자만 결제에 성공하도록 해야 합니다. 하지만 결제 프로세스를 떠올려 봅시다. 결제 전 상품의 재고부터 확인해야겠죠? 재고가 0개가 아니어야 하고, 상품의 실제 재고가 사용자가 구매하려는 재고의 갯수보다 많아야 합니다.

  1. 데이터베이스에서 상품을 조회합니다.
  2. 상품의 재고를 확인합니다.
  3. 재고를 감소시킵니다.

위의 케이스와 동일한 프로세스 입니다. 그리고 동일한 문제가 발생해요. A와 B의 요청이 동시에 발생했을 때 두 요청 모두 상품의 quantity 를 1 로 읽어옵니다. 상품을 구매하는 데 아무 문제가 없죠. 재고를 0으로 업데이트 시키고 끝이 납니다. 사용자 A와 B 모두 결제에 성공한 것으로 처리됩니다.

3.2. 배송 상태 업데이트

이 경우엔 케이스를 좀 더 세분화 하면 두 가지 케이스가 발생할 수 있을 것 같아요.

3.2.1. 동시에 상태를 수정하는 경우

배송 관리 시스템에서 같은 주문에 대해 서로 다른 관리자가 동시에 상태를 업데이트 하면 잘못된 배송 상태가 저장이 될 수 있어요.

주문 상태가 “입금 완료“ 인 주문이 있습니다. 관리자 A가 이 주문을 “상품 준비 중“으로 변경하기 직전에 다른 관리자 B가 이 주문을 “취소 처리됨“으로 변경하면 최종적으로는 A와 B의 의도와는 다르게 잘못된 상태가 저장될 수 있습니다. 이처럼 여러 사용자가 동시에 수정하는 작업을 진행할 때 동시성 이슈가 발생할 수 있어요.

3.2.2. 외부 API 호출과 상태 수정

배송 관리 시스템으로 굿스플로 라는 서비스를 이용했었어요. 굿스플로 API 를 사용하여 배송 상태를 자동으로 변경해주는 시스템이었어요. 하지만 굿스플로 API가 불안정해 내부 관리자들이 수동으로 배송 상태를 업데이트 할 때가 종종 있었는데요. 이 때도 동시성 이슈가 발생할 가능성이 있습니다.

시스템이 자동으로 굿스플로 API를 통해 배송 상태를 업데이트 하려는 순간 관리자가 배송 상태를 직접 수동으로 변경할 때, 둘 중 하나의 상태가 덮어씌워져서 예상하지 못한 배송 상태가 저장될 수 있습니다.

3.3. 포인트 적립 및 차감

사용자들이 앱에서 스트라바와 연동하면 주행 기록만큼 포인트가 적립되는 시스템이 있었습니다. 가끔 포인트가 덜 적립되었다는 사용자의 요청이 있었습니다. 반대로 포인트가 더 적립된 경우도 있었구요. 왜 이런 일이 발생했던 걸까요???

포인트 적립과 차감이 동시에 요청되어 처리


유저A는 10,000 포인트를 가지고 있습니다. 스트라바와 연동하여 5,000 포인트가 적립되어야 하는 상황에 A가 1,000 포인트를 사용하기로 합니다. 이 요청이 동시에 일어나면 어떤 일이 발생할까요?

임계 영역에 동시 접근

포인트 적립 스레드와 포인트 차감 스레드가 유저 A 의 포인트를 조회하는 메서드에 접근하게 됩니다. 두 스레드 모두 유저의 포인트 10,000원을 참조하게 됩니다.

포인트 적립 스레드 실행:

  • 10,000 포인트에 5,000 포인트를 적립
  • A 의 포인트는 15,000원으로 처리됨

포인트 차감 스레드 실행:

  • 10,000 포인트에 1,000 포인트를 적립
  • A 의 포인트는 9,000원으로 처리됨

결론

손실을 따져봅시다. 포인트가 정상적으로 순차 처리된 경우엔 초기 10,000P + 5,000P - 1,000P = 14,000P가 됩니다. 유저 A의 포인트는 결과적으로 14,000P 로 처리되어야 합니다.

적립 스레드가 커밋된 후 차감 스레드가 커밋 되면 유저의 최종 포인트는 9,000P 입니다. 유저는 5,000P 가 손실된 것이죠.

차감 스레드가 커밋된 후 적립 스레드가 커밋 되면 유저의 최종 포인트는 15,000P 입니다. 유저는 1,000P의 이득을 보았습니다.

마무리하며

백엔드 개발자로서 기술적 역량을 확장하기 위해서는 동시성 이슈 해결, 데이터베이스에 대한 심층적 이해, 성능 최적화 등 이런 영역들을 깊이있게 공부하는 것은 매우 중요한 부분이라고 생각합니다. 그 중에서 재고 시스템으로 알아보는 동시성이슈 해결방법 강의를 들으면서 동시성 이슈에 대해 공부해보았는데요.

강의 소개에 등장하는 재고 시스템 정합성 문제를 처음 봤을 때 이전의 경험들이 떠올랐어요. 그 당시의 상황들이 동시성 이슈 문제가 아니었을 수도 있지만 동시성 이슈가 발생할 여지가 있는 경우를 상상해보면서 정리해 보았습니다.

최대한 많은 케이스를 생각하다보니 동시성 이슈가 무엇인지 더 잘 알게 되었어요. 문제의 원인을 파악했으니 이를 해결할 수 있는 방법을 찾아보아야 하는데요. 다음 번에는 동시성 이슈 해결 방법에 대해 공부해 보고 정리한 내용을 공유해보겠습니다!
