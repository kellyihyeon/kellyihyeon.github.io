더 이상 방황은 그만! 프로젝트 요구사항 정의하기

💡 프로젝트 소개는 여기 에 있습니다. 먼저 읽어주시면 이해하기 더 쉬울 거예요!

농구 동호회 모임 통장 관리 서비스에 ‘이달의 입금 내역 Top3 조회하기’ API 가 있습니다. API 의 엔드포인트는 /account-transactions/deposits/top3 입니다. 그리고 account-transactions 테이블에는 accountId 데이터가 저장되어 있습니다.

해당 API 는 계좌가 하나인 것을 전제로 만들어졌습니다. 그런데 문득 이런 생각이 들었어요. “만일 활성화된 계좌가 하나가 아니라 여러 개라면 해당 API 로는 표현을 못하지 않나?”
특정 계정의 입금 내역을 요청한다면 엔드포인트에 accountId 가 포함이 되어야해요.

계좌를 하나로 고정할지 여러 개를 둘지 어떻게 해야하지? 지금은 하나로 두고 나중에 수정을 해야하나? 이 상황을 어떻게 풀어나갔는지 그 과정을 공유하려고 합니다! 😃

API 제공자는 언제 이러한 결정을 내려야 하는가?

  • 활성화된 계좌가 하나인 경우의 엔드포인트
    • /account-transactions/deposits/top3
  • 계좌가 여러 개인 경우의 엔드포인트
    • /account-transactions/{accountId}/deposits/top3

두 가지 케이스를 보면 엔드포인트 뿐만 아니라 데이터를 조회하는 코드도 수정이 되어야 합니다. 특정 accountId 를 필터링하는 조건을 추가해야해요.

요구사항이 바뀔 가능성을 고려하여 초기 설계 단계에서부터 설계하는 것이 좋을지, 기존의 API 구조를 유지하되 나중에 요구사항이 변경된다면 API 를 변경하는 게 좋을지 고민이 되었어요. 그러다가 이런 생각을 하게 됩니다. 요구사항이 변경되는 경우는 자주 발생하는데 이 모든 가능성들을 다 계산해서 초기 설계 단계에 전부 반영을 해야 하는 게 맞나? 애초에 변경될 가능성이 높은 것과 그렇지 않은 것을 어떻게 선별하지? 어디까지 미리 생각을 해둬야할까?

나는 왜 자꾸 방황하는가?

아는 개발자분에게 이러한 고민 거리를 털어놨더니 프로젝트 초기에 요구사항을 명확하게 정하지 않고 바로 개발부터 들어갔던 게 방황의 원인인 것 같다는 피드백을 받았습니다. 우선 요구사항이 명확하게 정의해둬야 초기 설계 단계에서 해당 사항들이 반영된 API가 설계가 되니까요. 그런데 좀 의아했습니다. 나는 분명히 요구사항을 정의했는데? 요구사항을 정의하지 않고 개발에 들어가는 게 가능한가? 왜 요구사항이 명확하지 않다는 거지?
제가 정의한 요구사항을 한번 같이 보시죠!

이게 요구사항 아니야?

클라이언트의 니즈를 반영한 요구사항은 다음과 같아요.

회원

  • 모임 통장 전체 내역 확인
  • 통장 잔액 확인
  • 이달의 회비 입금률 확인
  • 회비 미입금자 명단 확인
  • 이달의 입금/출금 내역 상위 3개 조회
  • 해당 연도의 월별 입금 및 지출액 확인
  • 이벤트 미입금자 명단 확인

관리자

  • 모임 통장에 입금된 돈을 입출금내역에 기록한다.
  • 모임 통장에서 출금된 돈을 입출금내역에 기록한다.
  • 이벤트를 생성한다.
  • 이벤트 신청자를 이벤트 참여자 관리 명단에 등록한다.

요구사항이 무엇인지부터 알아보자

요구사항은 시스템이 충족해야 할 기능과 조건에 대한 설명입니다. 즉, “무엇”을 해야 하는지에 대한 설명이라고 할 수 있어요.

개념만 봐서는 제가 정의한 요구사항도 요구사항 같아보이는데 뭐가 다른가 싶었어요. 그래서 다른 사람들은 요구사항을 어떻게 정의했는지 찾아보았습니다.

요구사항 정의서 예시

회원

  • 회원가입을 할 수 있다.
    • 회원가입 시에는 아이디, 이름, 비밀번호가 필요하다.
  • 소셜로그인으로 로그인이 가능하다.
    • 소셜로그인의 종류는 카카오, 깃헙, 애플만 허용한다.
  • 원하는 상품을 찾기 위해 키워드를 검색할 수 있다.
    • 추천 키워드를 5개 노출한다.

😢 네… 이제 그만 봐도 될 것 같습니다. 이렇게 비교를 하면서 보니 제가 정의했던 건 기능 목록에 불과했던 것 같아요. 음…그럼 요구사항을 다시 정의해볼까요?

요구사항 명확히 하기

case1

동호회 모임 통장은 이미 변경된 적이 있어요. 앞으로도 총무가 바뀌게 되면 모임 통장의 계좌는 언제든 변경될 수 있습니다. 여기에서 요구사항을 도출해볼까요?

  • 관리자는 계좌를 추가할 수 있다.
    • 계좌의 상태는 활성화/비활성화 로 구분한다.

case2

활성화 상태인 계좌가 여러 개라는 것은 계좌를 용도별로 두고 관리하겠다는 의미예요. 예를들면 회비용 통장과 행사용 통장이 각각이라는 의미입니다. 팀의 회원 수가 늘어나고 농구 외 외부사항 활동들이 빈번히 발생한다고 하더라도 통장을 분리해서 관리할 확률은 현저히 낮습니다. 그러므로 활성화 상태인 계좌가 여러 개가 될 경우까지 고려하지 않아도 된다고 판단했어요.

  • 관리자는 계좌의 상태를 변경할 수 있다.
    • 활성화된 계좌는 하나여야 한다.
    • 계좌를 활성화 상태로 변경할 때 나머지 계좌의 상태는 비활성화 상태가 된다.

왜 요구사항을 정의해야 해?

요구사항 정의서는 프로젝트의 네비게이션 역할을 합니다. 위에서 들었던 예시를 떠올려봅시다. 요구사항이 명확히 정의되어 있지 않으니 API 를 설계하는 것부터 시작해서 비즈니스 로직을 작성하는 것까지 혼란스러웠어요. 활성화된 계좌가 몇 개인지에 따라 API 와 비즈니스 로직이 달라지니까요.

저는 이 프로젝트를 혼자 진행하고 있지만 백엔드 개발자가 여러명이라고 가정해봅시다. 요구사항 정의서 없이 프로젝트를 시작하면 어떤 일이 벌어질까요? 아마…시작이 안되겠죠?

개발 가이드라인 제공

요구사항 정의서는 프로젝트의 방향성을 명확하게 설정해주기 때문에 팀이 같은 페이지를 바라볼 수 있게 해줍니다. 요구사항 정의서를 기준으로 개발자들이 일관된 방식으로 개발을 진행할 수 있기 때문에 개발 중 방향을 잃지 않게 해줍니다. 이는 혼선을 줄이고 개발 효율성을 높이는 데 도움을 줘요!

커뮤니케이션 비용 감소

요구사항을 문서화하면 이해관계자 간의 오해와 혼란을 줄일 수 있으며, 각자가 원하는 바를 명확히 이해할 수 있습니다.

마무리하며

프로젝트 초기에 브레인 스토밍을 하면서 요구사항 정의서를 작성하고 이를 토대로 설계를 마치고 개발에 착수한다. 생각해보면 아주 당연한 개발의 흐름인데요. 저는 이걸 반대로 하고 있었어요. 어떤 서비스를 만들지 정하고 개발할 때 필요한 테이블 설계를 해놓고 바로 개발에 착수했어요.

코드를 작성하면서 비즈니스 로직을 생각하니 이런 혼란스러운 일이 발생했던 거죠. 설계를 다시 해야 하는 일도 있었습니다. 😭 왜 자꾸 이런 일이 발생하는 것인지…

정말 감사하게도 ‘요구사항 정의’에 대한 피드백을 받게 되어 깨닫게 됐어요. 이 부분을 찾아보면서 제 개발 습관에 대해서도 알게 됐어요. 냅다 코드부터 작성하는 조급함, 기능을 빨리 구현하고 싶은 마음에서 비롯된다는 걸 깨닫게 된 시간이었어요.

프로젝트를 진행하면서 부족한 부분을 많이 느끼고 있는데요. 문제를 인지하고 공부하고 채워나가면서 어제보다 오늘 하나를 더 배운 사람이 되어 가고 있어요. 혹시나 저와 비슷한 고민을 하시는 분들께 조금이라도 도움이 되길 바라면서 고민 내용을 정리해 보았는데요. 앞으로도 다양한 문제들을 겪게 되겠죠? 이 문제들을 어떻게 풀어나갔는지 계속해서 공유해보록 하겠습니다!
